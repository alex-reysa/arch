# Inspecting Arch's Generated Output

This guide teaches a reviewer how to answer three questions about any file in an
Arch-generated backend:

1. **Which spec produced this file?** (traceability: artifact + entities + IR fragment)
2. **Will Arch overwrite this file?** (ownership: `ownership_kind` + `write_scope`)
3. **Why did Arch change this / report drift?** (`.arch/drift.json` + run reports)

Everything here is derived from the actual implementation. The two places truth
lives are the **per-file header** that the generator embeds at the top of every
output, and the **`.arch/` metadata** that the sync engine maintains alongside
the project. Both are described below with exact field names.

---

## 1. The generated-file header

Every generated file carries a traceability header. For TypeScript / JavaScript
output it is a `/** ... */` block; the same fields are emitted as `//` (Prisma),
`#` (YAML), `<!-- -->` (Markdown), or an `_arch` object (JSON). The TS form is
produced by `renderHeader(info, "ts")` in
`packages/arch-generator/src/generator.ts`.

### Example (`src/models/Post.ts`)

```ts
/**
 * Generated by Arch
 * artifact_id: tmpl.model.Post
 * entity_ids: [model.Post]
 * ownership_kind: generated_file
 * write_scope: whole_file
 * generator_id: arch.generator.v1
 * template_id: ts.model
 * ir_fragment_hash: 9f2c1ab4e7...c3
 */
```

> The literal values above are illustrative; only the **field names** are
> guaranteed. `artifact_id`, `entity_ids`, and `ir_fragment_hash` vary per file.

### Field reference

| Field | Meaning | Where it comes from |
|---|---|---|
| `Generated by Arch` | Marker line (`HEADER_MARK`). Its presence means Arch owns this file's header; do not hand-edit generated files. | `HEADER_MARK` constant |
| `artifact_id` | Stable logical identity of the artifact, e.g. `tmpl.model.Post`. This is the join key into `artifact-map.json` and `ownership.json`. | `FileSpec.artifactId` |
| `entity_ids` | The spec entity IDs this file was generated from, e.g. `[model.Post]` or a workflow plus its steps and guarantees. **This is your "which spec line" link** — cross-reference against `source-map.json`. | `FileSpec.entityIds` |
| `ownership_kind` | One of `generated_file`, `generated_region`, `extension_point`, `human_file`. See §3. | `HeaderInfo.ownershipKind` |
| `write_scope` | One of `whole_file`, `generated_region`, `stub_only`, `none`. Tells you whether Arch may overwrite this file. See §3. | `HeaderInfo.writeScope` |
| `generator_id` | The generator that produced the file. V1 is always `arch.generator.v1` (`GENERATOR_ID`). | `HeaderInfo.generatorId` |
| `template_id` | The specific template used, e.g. `ts.model`, `ts.workflow`, `ts.route`, `ts.guarantee-test`. | `HeaderInfo.templateId` |
| `ir_fragment_hash` | SHA-256 of the **source-position-stripped** canonical IR fragment used to render this file. An unrelated edit elsewhere in the spec shifts line numbers but must NOT change this hash — only a semantic change to *this* entity does. | `fragmentHash()` / `stripSourceFields()` |
| `stub_only` | Present and `true` only for extension points (`src/integrations/**`, `src/custom/**`). Indicates Arch wrote the file once and will never overwrite your edits. | added by `buildExtensionStub` |

JSON files (e.g. `package.json`, `tsconfig.json`) get the same data nested under
an `_arch` key via `jsonHeader(info)`: `_arch.generated_by`, `_arch.artifact_id`,
`_arch.entity_ids`, `_arch.ownership_kind`, `_arch.write_scope`,
`_arch.generator_id`, `_arch.template_id`, `_arch.ir_fragment_hash`.

## 2. The `.arch/` metadata directory

`arch init` scaffolds `.arch/` with subdirs `plans/`, `runs/`, `repair-history/`,
`locks/`, and `tmp/`. After a successful `arch apply` the following files exist.
Paths are resolved by `metadataPaths()` (`packages/arch-sync/src/metadata-store.ts`)
and `snapshotPaths()` (`packages/arch-sync/src/snapshots.ts`).

| File | Written by | Purpose |
|---|---|---|
| `ir.current.json` | `writeCurrent()` | The canonical IR compiled from the **current** `backend.arch`. The authoritative semantic intent. |
| `ir.previous.json` | `promoteStagedMetadata()` / `promoteCurrent()` | The IR baseline from the **last successful apply**. `arch plan` diffs current vs previous. Absent before the first successful apply. |
| `artifact-map.json` | `writeArtifactMetadata()` in `apply.ts` | One entry per generated file: `artifact_id → path + entity_ids`. The forward index from artifact identity to on-disk location. |
| `ownership.json` | `writeArtifactMetadata()` in `apply.ts` | Per-artifact ownership decision + a content hash. Drives drift detection and overwrite rules. |
| `source-map.json` | `arch check` (and apply staging) | Maps each spec entity to its byte span in `backend.arch`. Lets you walk `entity_ids` back to exact source lines. |
| `drift.json` | `arch check` (`runCheck`) | The latest drift report (see §4). |
| `plans/latest.plan.json` | `arch plan` | A copy of the most recent `SyncPlanV1` under a stable name; `arch apply` reads this for incremental sync. |
| `plans/<plan-id>.plan.json` | `arch plan` | The full read-only plan keyed by `plan_id`, plus a `.plan.md` human summary. |
| `runs/<id>/report.json` | `writeReports()` in `reports.ts` | The structured verification result for one apply/repair run (see §5). |

### `artifact-map.json`

Top level: `schema_version: "arch.artifact-map.v1"`, plus `ir_hash` (the
`canonical_hash` of the IR that produced it) and `generator_version`. Each entry:

| Field | Meaning |
|---|---|
| `artifact_id` | Matches the header's `artifact_id`. |
| `path` | Project-relative POSIX path of the generated file. |
| `entity_ids` | The spec entities that produced it. |
| `ir_fragment_hash` | Same fragment hash as the file header. |
| `generator_id` | e.g. `arch.generator.v1`. |
| `template_id` | e.g. `ts.model` (empty string if none). |

To answer "where is the file for `model.Post`?", grep `artifact-map.json` for the
entity id and read the `path`.

### `ownership.json`

Top level: `schema_version: "arch.ownership.v1"`. Each entry:

| Field | Meaning |
|---|---|
| `artifact_id` | Join key to `artifact-map.json`. |
| `ownership_kind` | `generated_file` \| `generated_region` \| `extension_point` \| `human_file`. |
| `write_scope` | `whole_file` \| `generated_region` \| `stub_only` \| `none`. |
| `content_hash` | SHA-256 of the file's full content **as Arch last wrote it**. The drift detector re-hashes the on-disk file and compares against this. |

### `source-map.json`

Written with `schema_version: "arch.source-map.v1"` and an `entries` array of
`SourceLocationIR` (`packages/arch-ir/src/source-map.ts`,
`packages/arch-ir/src/schema.ts`). Each entry:

| Field | Meaning |
|---|---|
| `id` | Source-location id. |
| `entity_id` | The spec entity (matches `entity_ids` in headers / artifact map). |
| `file` | Source file, normally `backend.arch`. |
| `start_line`, `start_column`, `end_line`, `end_column` | The exact span in the spec. |

**This is the bridge from a generated file back to a spec line.** Take an
`entity_id` from a file header, look it up here, and you have the line and column
range in `backend.arch` that produced it.

## 3. Ownership: will Arch overwrite this file?

Two fields govern overwrite behavior. They appear in the header AND in
`ownership.json`. The canonical definitions live in
`packages/arch-ir/src/schema.ts` (`OwnershipWriteScope`, `OwnershipKind`).

### `write_scope`

| Value | Arch's write rights | Typical files |
|---|---|---|
| `whole_file` | Arch may rewrite the **entire file** on every apply. Hand edits are lost and reported as drift. | models, validators, routes, workflows, runtime, config, tests |
| `generated_region` | Arch may patch only inside marked regions; text outside the markers is preserved. | (reserved for region-based artifacts) |
| `stub_only` | Arch writes the initial stub **once**, then never overwrites your edits. | `src/integrations/**`, `src/custom/**` extension points |
| `none` | Human-owned. Arch must never write. | hand-authored files Arch does not generate |

### `ownership_kind`

| Value | Meaning |
|---|---|
| `generated_file` | Fully Arch-owned; regenerated wholesale. Paired with `whole_file`. |
| `generated_region` | Arch owns only marked regions of an otherwise shared file. |
| `extension_point` | Arch seeds a stub for you to implement. Paired with `stub_only`; carries `stub_only: true` in the header. |
| `human_file` | Owned by a human; Arch never writes. Paired with `none`. |

**Rule of thumb:** if a file's header says `write_scope: whole_file`, treat it as
read-only — edit the spec, not the file. If it says `stub_only`, it is yours to
implement and Arch will leave it alone. In V1 the generator emits `generated_file`
/ `whole_file` for normal output and `extension_point` / `stub_only` for the
`src/integrations/**` and `src/custom/**` stubs.

### `src/custom/**` is human-owned

`src/custom/**` (and its `README.md`) is generated as `extension_point` /
`stub_only`. Arch writes these stubs once and **never overwrites your edits
afterward** — the generated `src/custom/README.md` states this directly. The
drift detector explicitly skips them: it only inspects artifacts whose ownership
is `generated_file` with `write_scope: whole_file`, so editing a `stub_only`
file never raises drift.

## 4. Drift: why did Arch flag this file?

`arch check` (`packages/arch-cli/src/commands/check.ts`) compares on-disk files
against `ownership.json` + `artifact-map.json` and writes the result to
`.arch/drift.json`. The detector lives in
`packages/arch-verifier/src/drift-detector.ts`. Output `schema_version` is
`arch.drift.v1`, with an optional `checked_ir_hash` (the canonical hash of the IR
checked against) and an `entries` array.

### Drift kinds (`DriftKind`)

| `kind` | Trigger |
|---|---|
| `generated_file_modified` | An Arch-owned (`generated_file` / `whole_file`) file's on-disk SHA-256 differs from the `content_hash` baseline — i.e. it was hand-edited. |
| `generated_file_missing` | A non-test artifact declared in `artifact-map.json` is no longer on disk. |
| `missing_generated_test` | A generated test file (`tests/{generated,guarantees,models,workflows}/**` or `*.test.ts`) declared in the map is missing. |
| `missing_guarantee_test` | A declared guarantee's generated test file is absent (grouped with `missing_generated_test`). |
| `guarantee_static_pattern` | Static scan of an on-disk workflow file matched a guarantee-violating shape — e.g. `notification_failure_does_not_rollback_post` appears to roll back a persisted post on notification failure. |

Each `DriftEntry` carries: `kind`, `artifact_id`, `path`, `entity_ids`, a
human `message`, and (for hash drift) `expected_hash` + `actual_hash`. So a single
entry tells you the file, the spec entities behind it, and exactly what changed.

### Category view (`by_category`)

`drift.json` also groups entries so tooling can consume by category without
re-deriving from `kind`:

- `artifact_hash` ← `generated_file_modified`
- `missing_artifact` ← `generated_file_missing`
- `missing_guarantee_test` ← `missing_generated_test` + `missing_guarantee_test`
- `guarantee_static_pattern` ← `guarantee_static_pattern`

`arch check` exit codes: `0` no drift, `1` drift detected (report written), `2`
invalid input or metadata corruption.

## 5. Run reports: `runs/<id>/report.json`

Each verification run (during `arch apply` / `arch repair`) writes a structured
report via `writeReports()` in `packages/arch-verifier/src/reports.ts`. Shape:

| Field | Meaning |
|---|---|
| `schema` | `"arch.verification-report.v1"`. |
| `run_id` | The run id; matches the `runs/<id>/` directory name. |
| `passed` | Whether every step passed. |
| `failure_reason` | A string, or `null` on success. |
| `steps[]` | Per-step records: `name`, `passed`, `duration_ms`, and truncated `stdout_tail` / `stderr_tail` (last 4096 chars). |

A human-readable `report.md` is written alongside it (and mirrored as
`verification-report.json` / `verification-report.md`). Metadata is promoted to
the baseline **only after a run passes**: `ir.current.json → ir.previous.json`,
and staged `artifact-map.json` / `ownership.json` / `source-map.json` are renamed
into place (`promoteStagedMetadata`). A failed run leaves the previous baseline
untouched, so drift checks still compare against the last known-good state.

## 6. Hands-on walkthrough

Run the SocialFeed demo from the README, then inspect the headers and metadata.
In this repo, invoke the CLI as `pnpm exec tsx packages/arch-cli/src/main.ts <cmd>`
(or use the built `arch` bin).

```sh
mkdir demo && cd demo
cp ../examples/social-feed/v1/backend.arch .

arch init                          # scaffold .arch/ + src/custom
arch parse --emit-ir backend.arch  # writes .arch/ir.current.json
arch plan                          # read-only SyncPlanV1
arch apply                         # generate, install, verify, promote
```

Inspect a generated file's traceability header:

```sh
# What spec produced src/models/Post.ts, and may Arch overwrite it?
head -n 12 src/models/Post.ts
```

Inspect the metadata:

```sh
# Forward index: artifact_id → path + entity_ids
cat .arch/artifact-map.json

# Ownership + content hashes that drive drift detection
cat .arch/ownership.json

# entity_id → exact line/column span in backend.arch
cat .arch/source-map.json

# The IR baseline diff is over these two
cat .arch/ir.current.json
cat .arch/ir.previous.json
```

Trace a generated file back to a spec line:

1. `head -n 12 src/models/Post.ts` → read `entity_ids: [model.Post]`.
2. Grep `model.Post` in `.arch/source-map.json` → read `start_line` / `end_line`.
3. Open `backend.arch` at that span — that is the spec text Arch compiled.

Confirm a `stub_only` file is left alone:

```sh
head -n 12 src/custom/README.md   # write_scope: stub_only, stub_only: true
echo "// my edit" >> src/custom/<Something>.ts
arch check                        # no drift: stub_only files are skipped
```

See drift and the run report in action:

```sh
rm tests/guarantees/no_unsanitized_html_persisted.CreatePost.test.ts
arch check                        # missing_generated_test drift, exit 1
cat .arch/drift.json              # entries[] + by_category

arch repair                       # regenerates from IR, re-verifies
cat .arch/runs/*/report.json      # schema arch.verification-report.v1
arch check                        # clean, exit 0
```

## Quick reference: answering the three questions

- **Which spec line produced this file?** Read `entity_ids` from the file header
  (or `artifact-map.json`), then look those ids up in `source-map.json` for the
  exact span in `backend.arch`.
- **Will Arch overwrite this file?** Read `write_scope`: `whole_file` = yes,
  every apply; `stub_only` = no, only the first stub; `none` = never. `src/custom/**`
  is `stub_only` and human-owned.
- **Why did Arch change this / flag it?** For drift, read `.arch/drift.json`
  (`kind` + `message` + `expected_hash`/`actual_hash`). For an apply, the
  `ir_fragment_hash` changing between baselines means the entity's semantic intent
  changed; `runs/<id>/report.json` records whether the resulting run verified.
